CAP Theorem

  • Consistency
    every read receives the most recent write (a single-copy illusion)
  • Availability
    every request receives a non-error response (even if some nodes are down)
  • Partition tolerance
    the system keeps operating even when network links are lost
Reveal one letter at a time. C is shorthand for linearizability; A means every request to a non-failing node gets a response - the precise Gilbert & Lynch definitions are the back-pocket answer if someone pushes back. Avoid saying "pick 2 of 3": partitions are not optional, so the real choice is the next slide.

Rule: In the presence of a network partition, a distributed system must choose either Consistency or Availability.

Point at the drifting values while partitioned - that drift IS the C-vs-A choice: serve stale (A) or refuse to answer (C).

The Two Generals Problem

Two generals are planning to attack a city. They can only communicate via messengers, who may be captured.

How can both generals be sure they attack at the same time, given that messages might not arrive?

The loop shows all three outcomes in random order: message captured, ACK captured, and both delivered - and even when everything arrives, G2 still can't know G1 got the ACK. Name the infinite regress out loud: every ack would need its own ack.

No protocol can guarantee that two parties agree to act over an unreliable channel.

Every additional ack raises confidence - never certainty.

Deliberately "no protocol", not "no deterministic way" - randomized protocols can't solve it either. If asked "doesn't TCP solve this?": TCP retransmits, but the regress applies to the final ack - the application layer still can't reach common knowledge.

Definition

An operation is idempotent if performing it multiple times produces the same effect as performing it once.


f(f(x)) = f(x)

Failures and retries are inevitable

  • Networks can fail between send and acknowledge
  • Clients and load balancers retry
  • Servers can crash mid-processing

Exactly-once delivery?

A fairy tale.

Reality: You only ever get:

  • at-most-once -> messages may be lost
  • at-least-once -> messages may be duplicated

Exactly-once effect

We can't guarantee exactly-once delivery,

but we can design for exactly-once effect using idempotency.

at-least-once delivery + idempotency = exactly-once effect

PACELC Theorem

Daniel Abadi extended CAP to include trade-offs when no partition occurs:

If P (partition) -> A or C
Else (ok) -> Latency or Consistency
Re-anchor: back to the C-or-A choice from the opening. Abadi's point: the trade-off doesn't disappear when the network is healthy - you just pay in latency instead of availability.

Even without partitions, there’s another trade-off

Waiting for replicas to acknowledge takes time

This is the “ELC” part of PACELC.

Imagine two check-in desks for the same flight

  • network failure -> choice: keep assigning seats, risk duplicates (A) or freeze until synced (C)
  • no failure -> each assignment can be: fast but from a cached seat map (L) or slow but from a confirmed seat map (C)

Real-world examples

  • Amazon Dynamo / Cassandra -> prioritize Availability + Low Latency
  • Google Spanner -> prioritize Consistency, accept more latency

False Dichotomy:

Consistent vs. Inconsistent

Different consistency levels

  • Strong consistency -> Data "converges" immediately
  • Eventual consistency -> Data converges... eventually
  • Accidental consistency -> Data converges... maybe

All are “consistent” - just in different ways.

Eventual consistency

If no new updates occur, all replicas will eventually converge to the same state.

It’s a trade-off: we get availability and speed at the cost of temporary disagreement.

Let one burst play out: amber pulse = a write lands on one replica, versions diverge (stale replicas tint amber), gossip dots carry the newest version around, then the teal "converged" ring. Key phrase: "eventually" is a guarantee about the destination, not the journey.

Why this makes sense

  • Networks are slow and unreliable
  • Waiting for everyone to agree slows everything down
  • So we let nodes respond now and sync later

-> Fast now, consistent later

Where we’re heading:

Transactional Outbox + Change Data Capture

Transactional Outbox Pattern

Write the event to a local outbox table as part of the same transaction.


@Transactional
public void createOrder(Order order) {
    orderRepository.save(order);
    outboxRepository.save(new OutboxEvent("OrderCreated", order.id()));
}

Then a separate process reads the outbox and publishes the events

Trivial Publisher


while (true) {
    for (OutboxEvent event : outboxRepository.findUnpublished()) {
        eventBus.publish(event.toDomainEvent());
        event.markAsPublished();
    }
}

Reliable, asynchronous, and decoupled event publishing.

...or skip the polling and let the database stream its own changes: Change Data Capture

CDC in Action

  • Tools like Debezium rely on db replication protocols
  • Each committed change becomes an event
  • No need to modify application logic

-- Database change -->
INSERT INTO outbox (...) VALUES (...)

-- Debezium -->
Produces Kafka event: OrderCreated { orderId: 42 }

Outbox + CDC: the deal

  • ✓ atomic writes - no dual writes, no lost events, no distributed transactions
  • ✓ events stream out automatically
  • ✗ extra tables and processes
  • ✗ duplicates still happen -> consumers must be idempotent

OSS Support

  • outboxd - a lightweight event relay daemon for PostgreSQL implementing the transactional outbox pattern with WAL-based replication
  • fencepost - distributed locking, leader election, and message queuing backed entirely by PostgreSQL

Time

The clock starts at real wall time, then quietly accelerates - by the end of this sentence it's running several times too fast and the multiplier is turning red. Punchline: every clock in your fleet is doing this; yours just lies more slowly.

Why clocks go backwards

  • NTP (Network Time Protocol) adjustment
  • VM snapshot restores and live migrations
  • Leap seconds
  • Manual time correction by an admin
Leap-second aside: Google and AWS "smear" the extra second over many hours - clocks deliberately run at the wrong RATE so they never step. A perfect on-theme trade-off: accept rate error to avoid step error. If challenged on live migrations: the backward step comes from the one-shot hypervisor time sync AFTER the move (e.g. VMware Tools post-vMotion corrects a fast guest backwards) or a destination host whose clock is behind the source - the migration pause itself jumps forward.

A monotonic clock always moves forward - never backward.

But most system clocks are not monotonic.

Monotonic clocks to the rescue

  • In Java: System.nanoTime()
  • In Linux: CLOCK_MONOTONIC

They never go backward, but don’t represent 'real' time.


Measuring time: from Java to kernel and back

In distributed systems, time is… an illusion.

Monotonic clocks fix one machine

Across machines you’re back to wall clocks. Two servers record an event:

  • Server A: event at 12:00:00.100
  • Server B: event at 12:00:00.090

Which happened first?

The pivot of the whole section: monotonic clocks fix ONE machine, but each machine still has its own imperfect clock, and network delay makes "now" ambiguous between them. Replication, logs and conflict resolution all need an answer to "what came first" - and wall clocks can't give one. So we stop asking "when" and start asking "what caused what".

Happened-before relationship

Instead of wall-clock time, we use causal order:

  • same process: a -> b if a comes first
  • message: a -> b if a is the send and b the receive
  • transitivity: a -> b and b -> c ⇒ a -> c

We care about ordering of events, not their timestamps.

The canvas demonstrates rules 1 and 2: events popping along each process line (same-process order), then the message arrow from P1 to P2 (send -> receive, note the max(1,2)+1 counter). The amber pulse pair at the end is the punchline: two events with NO path between them are concurrent - no causal order exists, and no protocol can recover one.

Logical clocks

  • Lamport clocks -> an order consistent with causality (one-way: a -> b ⇒ L(a) < L(b), but not the reverse)
  • Vector clocks -> can additionally detect that events are concurrent

They don’t measure real time - they encode an order consistent with cause and effect.

Careful wording: Lamport timestamps don't "track" causality - concurrent events still get ordered timestamps (the amber pair on the previous slide would, too). They only promise the one-way implication. Detecting concurrency is exactly the gap vector clocks close.

Real time vs. logical time

  • Real time -> what your watch shows
  • Logical time -> what the system can prove happened first

Distributed systems live in logical time.

Google’s Spanner uses special hardware clocks (TrueTime API)

But even then, they include an uncertainty window - because perfect time doesn’t exist.

Summary

  • Each node has its own imperfect sense of time
  • We can't rely on timestamps for ordering
  • We use logical or causal time to reason safely

Time is relative - especially in distributed systems

Dancing with uncertainty

You can’t eliminate uncertainty - you learn the steps:

  • Assume at-least-once -> make every consumer idempotent
  • Never dual-write -> outbox + CDC
  • Never order events by wall clock -> use causal order
  • Pick your trade-offs explicitly -> A vs C, latency vs consistency
The payoff of the title: you never get rid of uncertainty - you choreograph around it. One actionable line per chapter of the talk; this is the slide people should photograph. If someone quotes Spanner against rule 3: TrueTime is the exception that proves it - wall-clock ordering is safe only when clock uncertainty is explicit, bounded, and waited out; commodity NTP gives you none of that.
Using common sense is the ultimate Best Practice™.
